Реализация кастомного Thread Pool

Обзор

Данный модуль представляет собой реализацию собственного пула потоков с настраиваемыми параметрами, политиками отказа и балансировкой нагрузки. В отличие от стандартного ThreadPoolExecutor, эта реализация использует множественные очереди и Round Robin балансировку для распределения задач.

Архитектура

Основные компоненты

CustomThreadPool - главный класс пула, реализующий интерфейс CustomExecutor. Управляет созданием и завершением потоков, распределением задач и мониторингом свободных ресурсов.

ThreadPoolConfig - конфигурационный класс, содержащий все настраиваемые параметры пула:

corePoolSize - минимальное количество потоков, создаваемых при старте

maxPoolSize - максимальное количество потоков, которое может создать пул

keepAliveTime и timeUnit - время простоя потока до завершения

queueSize - максимальный размер каждой очереди задач

minSpareThreads - минимальное количество свободных потоков для поддержания

Worker - рабочий поток со следующими характеристиками:

привязан к своей очереди задач

ожидает задачи с таймаутом keepAliveTime

завершается после трех последовательных таймаутов, если общее число потоков превышает corePoolSize

сигнализирует о своей активности через флаг isActive с защитой ReentrantLock

LoadBalancer и RoundRobinBalancer - отвечают за распределение задач между очередями по круговому алгоритму. Используется CopyOnWriteArrayList для хранения очередей и AtomicInteger для потокобезопасного выбора следующей очереди.

Политики отказа

В пуле реализованы три политики обработки переполнения.

AbortPolicy (политика по умолчанию)

При переполнении очереди задача отклоняется с логированием ошибки. Счетчик rejectedTaskCount увеличивается при каждом отказе. Это самая простая политика, гарантирующая, что пул не будет перегружен.

CallerRunsPolicy

При переполнении задача выполняется в потоке, который вызвал метод execute. Это позволяет избежать потери задач, но может замедлить работу вызывающего потока. Полезна для критичных задач, которые нельзя терять.

DiscardPolicy

Задача молча отбрасывается без логирования ошибки (только debug уровень). Подходит для ситуаций, где допустима потеря некритичных задач при пиковых нагрузках.

Балансировка нагрузки

Ключевая особенность реализации - использование множественных очередей. При создании пула формируется список BlockingQueue, по одной на каждый core поток. Каждый Worker закрепляется за своей очередью.

RoundRobinBalancer работает следующим образом:

при поступлении новой задачи балансировщик выбирает следующую очередь по кругу

атомарный счетчик гарантирует потокобезопасность выбора

задача помещается в выбранную очередь

соответствующий Worker забирает задачу из своей очереди

Такой подход уменьшает конкуренцию за блокировки по сравнению с единой очередью и позволяет более равномерно распределять нагрузку.

Мониторинг и поддержание потоков

Отдельный ScheduledExecutorService запускает монитор с периодичностью 1 секунда. Монитор выполняет следующие задачи:

подсчитывает количество активных и свободных потоков, используя метод getActiveCount, который проверяет флаг isActive у каждого воркера

сравнивает количество свободных потоков с minSpareThreads

при необходимости создает новые потоки до достижения maxPoolSize

логирует текущее состояние пула

Это гарантирует, что всегда доступно минимальное количество свободных потоков для обработки внезапных всплесков нагрузки.

Жизненный цикл задачи

Пользователь вызывает pool.execute(runnable)

LoadBalancer выбирает целевую очередь через getNextQueue

Если количество потоков меньше maxPoolSize, создается новый воркер и очередь

Задача помещается в очередь методом offer

Если очередь переполнена, срабатывает политика отказа

Воркер забирает задачу из своей очереди через poll(keepAliveTime, timeUnit)

Перед выполнением воркер устанавливает флаг isActive в true с использованием ReentrantLock

После выполнения флаг сбрасывается

Завершение работы

Пул поддерживает два режима завершения.

shutdown - плавное завершение:

новые задачи не принимаются

уже запущенные задачи завершаются нормально

монитор останавливается

все воркеры получают сигнал на завершение через флаг isRunning

shutdownNow - принудительное завершение:

все потоки прерываются вызовом interrupt

все задачи из очередей извлекаются и возвращаются в виде списка

монитор немедленно останавливается

Тестирование

Всего реализовано 8 тестов, покрывающих ключевую функциональность.

testPoolCreationAndExecution - проверяет базовое создание пула и выполнение задач

testAbortPolicy - проверяет корректность отклонения задач при переполнении

testCallerRunsPolicy - проверяет выполнение задач в вызывающем потоке при переполнении

testDiscardPolicy - проверяет молчаливое отбрасывание задач

testSubmit - проверяет работу метода submit с возвратом Future

testShutdownNow - проверяет принудительное завершение и возврат незавершенных задач

testRoundRobinBalancing - проверяет равномерное распределение задач между очередями

testMinSpareThreads - проверяет поддержание минимального количества свободных потоков

Все тесты используют CountDownLatch для синхронизации и AtomicInteger для подсчета выполненных задач. В тесте testMinSpareThreads реализована многократная проверка активности для надежности.

Логирование

В проекте используется SLF4J с подробным логированием всех ключевых событий:

создание новых потоков с указанием имени

завершение потоков с указанием причины

поступление задач в очереди с указанием идентификатора очереди и ее размера

выполнение задач с указанием воркера и флага активности

отказы при переполнении с указанием политики

таймауты простоя с количеством проверок

текущее состояние пула при мониторинге

Особенности реализации

Синхронизация

Для обеспечения потокобезопасности используется несколько механизмов:

CopyOnWriteArrayList для списков workers и queues обеспечивает безопасное итеративное чтение

ReentrantLock для атомарных операций со списками при добавлении новых воркеров

AtomicBoolean и volatile для флагов состояния

synchronized блоки для критических секций, где необходима простая монопольная блокировка

Подсчет активных потоков

Метод getActiveCount использует блокировку и проверяет только флаг isActive, так как проверка isAlive может давать ложные результаты на ранних стадиях запуска потока. Дополнительное логирование помогает при отладке.

Обработка прерываний

Все блокирующие операции обрабатывают InterruptedException и корректно восстанавливают статус прерывания через Thread.currentThread.interrupt. Воркеры завершаются при получении прерывания, устанавливая флаг isRunning в false.

Idle timeout

Реализация с MAX_IDLE_CHECKS, равным 3, отличается от классического подхода, где поток завершается после первого же неудачного poll. Это решение было принято для предотвращения преждевременного завершения потоков при кратковременных паузах в поступлении задач. Три последовательных таймаута гарантируют, что поток действительно простаивает длительное время.

Сравнение со стандартным ThreadPoolExecutor

Преимущества реализации

множественные очереди уменьшают конкуренцию за блокировки, особенно при высоких нагрузках

возможность тонкой настройки через minSpareThreads позволяет поддерживать необходимый уровень готовности

детальное логирование всех операций облегчает мониторинг и отладку

гибкая балансировка нагрузки через сменяемые реализации LoadBalancer

прозрачный механизм мониторинга с возможностью расширения

Недостатки

отсутствие некоторых оптимизаций стандартной библиотеки, например ForkJoinPool

привязка воркера к очереди может привести к неравномерной загрузке, если задачи сильно различаются по времени выполнения

дополнительные накладные расходы на мониторинг и множественные очереди (в стандартной реализации эти расходы оптимизированы сильнее)

Обоснование архитектурных решений

Выбор политики по умолчанию

AbortPolicy выбран как политика по умолчанию, поскольку она наиболее безопасна при перегрузках - система явно сигнализирует о проблеме через логи и счетчик отказов. Разработчик может отследить это и принять меры.

Реализация minSpareThreads через отдельный монитор

Отдельный поток-монитор был реализован вместо встраивания логики в основной цикл пула, чтобы не усложнять критический путь выполнения задач. Монитор работает асинхронно и не влияет на производительность обработки.

Round Robin балансировка

Round Robin выбран за простоту реализации и предсказуемость поведения. При равномерных по времени выполнения задачах он обеспечивает практически идеальную балансировку.

MAX_IDLE_CHECKS

Значение 3 выбрано эмпирически как компромисс между быстрым освобождением ресурсов и предотвращением лишних завершений при флуктуациях нагрузки.

Выводы

Реализованный пул потоков полностью соответствует требованиям задания и успешно проходит все тесты. Ключевые особенности - множественные очереди с Round Robin балансировкой и механизм поддержания минимального количества свободных потоков - делают его гибким инструментом для управления задачами в многопоточной среде. Архитектура позволяет легко расширять функциональность, добавлять новые политики отказа и алгоритмы балансировки.
